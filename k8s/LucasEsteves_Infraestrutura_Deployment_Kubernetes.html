Arquitetura proposta

A solucao foi organizada para atender ao enunciado do trabalho, separando os recursos entre aplicacao, persistencia e monitoramento. O projeto roda no Minikube, com manifests centralizados na pasta k8s, e a exposicao externa foi limitada ao Grafana. Os demais servicos permanecem internos no cluster.

Estrutura utilizada

O que cada arquivo faz

Resumo rapido dos principais arquivos da entrega:

Arquivo Funcao
kustomization.yamlreune todos os manifests e gera os ConfigMaps dos dashboards
namespace.yaml e secrets.yamlcriam o namespace e centralizam credenciais
application/backend.yaml e frontend.yamldefinem frontend e backend com seus Deployments e Services
database/mongodb.yaml e rabbitmq.yamlcada arquivo concentra PVC + Deployment + Service do seu servico
monitoring/prometheus.yamlconfigura coleta de metricas, PVC, Deployment, Service e RBAC
monitoring/grafana.yamlconfigura datasources, dashboards, Deployment e o unico NodePort
monitoring/loki.yamldefine logs centralizados com ConfigMap + PVC + Deployment + Service
monitoring/kube-state-metrics.yamlexpoe metricas do estado do cluster para o Prometheus
dashboards/*.json e scripts/stress-test.ps1dashboards do Grafana e script para teste de carga

Em resumo: application = app, database = persistencia e mensageria, monitoring = observabilidade. Os PVCs ficam junto de MongoDB, RabbitMQ, Prometheus e Loki.

Organizacao da arquitetura no cluster

Com base nessa estrutura, os componentes foram distribuidos no cluster da seguinte forma:

  • backend: Deployment com 4 replicas e Service ClusterIP
  • frontend: Deployment com 1 replica e Service ClusterIP
  • mongodb: Deployment com 1 replica, Service ClusterIP e PVC
  • rabbitmq: Deployment com 1 replica, Service ClusterIP e PVC
  • prometheus: Deployment com 1 replica, Service ClusterIP e PVC
  • grafana: Deployment com 1 replica e Service NodePort
  • kube-state-metrics: coletor de metricas do cluster
  • loki: agregacao de logs

Regra de exposicao externa

  • Externo: apenas grafana
  • Interno: backend, frontend, mongodb, rabbitmq, prometheus e loki

Pre-requisitos

  • Docker Desktop

Passo a passo de execucao

1. Iniciar o cluster

minikube start
minikube status

2. Aplicar os manifests

kubectl apply -k k8s
kubectl wait --for=condition=ready pod --all -n questionario --timeout=300s

Observacao: esse passo pode levar alguns minutos, principalmente por causa do tempo de inicializacao do RabbitMQ e das validacoes de readiness.

3. Validar os recursos

kubectl get pods -n questionario
kubectl get svc -n questionario
kubectl get pvc -n questionario
  • backend com 4 pods Running
  • grafana com NodePort
  • backend, frontend, mongodb, rabbitmq e prometheus com ClusterIP
  • PVCs de MongoDB, RabbitMQ, Prometheus e Loki em estado Bound

4. Acessar a aplicacao

Para atender aos requisitos do trabalho, frontend e backend estao publicados internamente no cluster por meio de Service ClusterIP. Por isso, para acessar a aplicacao localmente durante a demonstracao, e necessario utilizar kubectl port-forward em dois terminais separados.

Abra um terminal para o backend e execute:

kubectl port-forward svc/backend 5000:5000 -n questionario

Depois abra outro terminal para o frontend e execute:

kubectl port-forward svc/frontend 8080:80 -n questionario

Abrir no navegador:

  • Frontend: http://localhost:8080
  • Backend: http://localhost:5000/swagger

Usuario administrativo padrao:

  • email: admin@questionario.com
  • senha: Admin@123

6. Executar o stress test

Antes de iniciar o teste, manter o backend acessivel localmente em outro terminal com o comando abaixo:

kubectl port-forward svc/backend 5000:5000 -n questionario

Com o port-forward ativo, executar o script de carga:

.\k8s\scripts\stress-test.ps1
.\k8s\scripts\stress-test.ps1 -Requests 5000 -Concurrent 100
antes do teste de stresse depois

Monitoramento configurado

Prometheus

  • coleta metricas do Kubernetes
  • coleta metricas da API em /metrics
  • usa PVC para persistencia dos dados

Grafana

  • fica acessivel fora do cluster via NodePort
  • usa Prometheus como fonte principal de metricas
  • possui dashboards com CPU por pod, memoria por pod, pods ativos, latencia HTTP, total de requisicoes e erros 4xx e 5xx

Loki

  • recebe logs estruturados do backend
  • pode ser consultado via Grafana

Com esse conjunto de ferramentas, o projeto atende nao apenas ao requisito de deploy em Kubernetes, mas tambem ao requisito de observabilidade, permitindo acompanhar metricas e logs do ambiente de forma centralizada.

Pipeline de entrega

O projeto possui pipeline em GitHub Actions para restore, build, testes, build das imagens Docker e push para o Docker Hub.

Observacao importante
  • como o cluster usado e o minikube local, o deploy no cluster nao acontece automaticamente no GitHub hospedado
  • a entrega foi organizada como build/test/push automatico e deploy local com kubectl apply -k k8s
  • se necessario, isso pode evoluir para deploy automatico usando self-hosted runner

A imagem abaixo registra a execucao do pipeline, servindo como evidencia de que o fluxo de integracao continua foi configurado para validar e publicar a aplicacao.

Essa evidencia complementa a entrega mostrando que o projeto tambem possui automacao de build, testes e publicacao das imagens.
